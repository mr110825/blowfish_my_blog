# 技術的デジャブ回避メモ帳

技術的デジャブを回避するためのメモや備忘録を蓄積する技術ブログです。  
「あれ、これ前にも調べたな...」を防ぎ、効率的な技術学習と開発をサポートします。

## プロジェクト構成

- **Posts** (`content/posts/`): 技術記事・ハウツー・チュートリアル
- **Scraps** (`content/scraps/`): 学習メモ・備忘録・スクラップメモ
- **フレームワーク**: Hugo + Blowfishテーマ
- **デプロイ**: GitHub Pages

---

## 基本操作

### 開発サーバー起動

```bash
# 通常起動
hugo server

# 下書き含む（draft: trueの記事も表示）
hugo server -D
```

### 新規記事作成

```bash
hugo new posts/<記事のタイトル>.md
```

**注意**: 上記コマンドでは、記事のフロントマターに`id`（Unixタイムスタンプ）が自動付与されますが、**ディレクトリ名にはIDが反映されません**。

#### ディレクトリ名にIDを含める方法

**推奨**: 記事作成後、一括リネームスクリプトを実行

```bash
# IDが反映されていないディレクトリを一括リネーム
python3 rename_dirs_with_id.py
```

このスクリプトは：
- `content/posts`と`content/scraps`内の全ディレクトリを検索
- 各`index.md`からIDを取得
- `{ID}_ディレクトリ名`形式にリネーム（例: `test-memo` → `1759800890_test-memo`）
- `git mv`を使用してGit履歴を保持

詳細は [`claude_memo/20251007_hugo_id_system_implementation.md`](claude_memo/20251007_hugo_id_system_implementation.md) を参照してください。

## 記事品質保証ルール

### 基本方針

本ブログは**実用的で信頼できる技術情報**を提供することを目的とします。すべての投稿記事は以下の品質基準を満たす必要があります。

### 必須要件（投稿前チェックリスト）

#### ✅ メタデータ設定
- [ ] 適切なタイトル（具体的で検索しやすい）
- [ ] 投稿日時の設定
- [ ] 関連するタグの設定（2-5個推奨）
- [ ] `draft = false` に設定

#### ✅ 構成・読みやすさ
- [ ] **冒頭3行以内**で記事の目的・内容を明示
- [ ] **見出し階層**が適切に設定されている（H2→H3→H4の順）
- [ ] **段落は3-5行程度**に分割されている
- [ ] **コードブロック**には適切な言語指定とコメント
- [ ] **箇条書きや表**を効果的に活用
- [ ] **具体例と全体像**のバランスが取れている

### 内容品質基準（5つの評価軸）

#### 1. 防御力（信頼性確保）
- **主観的表現の使用**: 「〜は私の環境では効果的でした」
- **コンテキストと制約の明示**: 前提条件、対象読者、環境の明記
- **根拠と出典の提供**: 参考記事・公式ドキュメントへのリンク必須
- **限界と注意点の記載**: 対応できない場合や制限事項を明記

#### 2. 思考整理力（論理構造）
```markdown
## はじめに（問題提起・背景）
## 解決方法・手順
## 実際の例・コード
## つまずきポイントと対処法
## まとめ・次のステップ
```

#### 3. 実践応用性（即座に活用可能）
- **環境構築から完了まで**: セットアップ手順の詳細記載
- **コマンド例と実行結果例**: コピペで動作するコード
- **失敗例とその解決策**: よくあるエラーへの対処法

#### 4. 構成と読みやすさ
- 冒頭3行で読者の興味を引く
- 短い段落と適切な見出し階層
- 視覚的要素（箇条書き、表、強調）の効果的活用

#### 5. コミュニケーション力（読者配慮）
- **読者の立場に立った説明**: 専門用語の簡潔な説明併記
- **共感的な表現**: 個人的な経験や感想を織り込む

### 記事種別ごとの特別要件

#### 📝 ハウツー・チュートリアル記事
- [ ] **実行可能なコード例**（コピペで動作する）
- [ ] **スクリーンショット**または**実行結果例**
- [ ] **所要時間の目安**
- [ ] **前提知識レベル**の明示

#### 🔧 ツール・ライブラリ紹介記事
- [ ] **インストール方法**
- [ ] **基本的な使用例**
- [ ] **他のツールとの比較**（可能であれば）
- [ ] **使用場面・向いている用途**

#### 📚 学習メモ・備忘録記事
- [ ] **学習の動機・背景**
- [ ] **重要ポイントの整理**（表形式推奨）
- [ ] **実践で使えるコマンド・コード例**
- [ ] **関連技術へのリンク**

#### 🔍 トラブルシューティング記事
- [ ] **発生状況の詳細**（環境・操作手順）
- [ ] **エラーメッセージの全文**
- [ ] **解決手順**（試行錯誤のプロセス含む）
- [ ] **根本原因の説明**
- [ ] **予防策**

### 禁止事項

#### ❌ 内容面での禁止事項
- 検証していない情報の記載
- 他者の記事の無断転載
- セキュリティリスクのある手法の推奨
- 個人情報・機密情報の含有
- 誤解を招く断言的な表現

#### ❌ 形式面での禁止事項
- 見出しなしの長文
- コードブロックなしのコード記載
- 改行のない長い段落（6行以上）
- リンク切れの参考記事
- 動作確認していないコード例

### 品質チェック手順

#### 投稿前セルフチェック
1. **構成チェック**: 見出し構造・段落分割・視覚的要素
2. **内容チェック**: 事実確認・動作確認・参考リンク確認
3. **読者視点チェック**: 前提知識・説明の過不足・実用性
4. **文章チェック**: 誤字脱字・表現の適切性・リンク動作

#### 定期レビュー
- **月1回**: 過去記事の情報更新チェック
- **四半期1回**: タグ・カテゴリの整理
- **半年1回**: 評価基準での品質監査

### 例外規定

#### スクラップメモの場合
スクラップメモ（`content/scraps/`）については、以下の簡易基準を適用：
- 個人の学習記録として**主観的記述OK**
- **後から参照しやすい構造**を重視
- **実用性**と**整理性**を最優先
- 完全性よりも**記録の継続性**を重視

## 記事評価基準（詳細）

記事の品質を客観的に評価するため、以下の5つの観点で0.0～5.0の評価を行います。

### 評価軸の詳細

#### 防御力 (0.0-5.0)
記事が批判や反論に対してどれだけ耐性を持っているかを評価します。

**5.0**: 完璧な防御力。主観的表現と限定的な主張を適切に用い、コンテキストと限界を明示し、実体験と具体例が豊富で根拠と出典が明確。批判を先取りする構成で異なる立場への配慮が行き届いている。

**3.0**: 標準的な防御力。部分的に主観や限界を示しているが、一部に断言的な表現や根拠不足がある。

**1.0**: 非常に弱い防御力。断言と一般化が目立ち、コンテキストや根拠がほぼない。

#### 思考整理力 (0.0-5.0)
記事が著者の思考プロセスを整理し、知識を構造化して伝えているかを評価します。

**5.0**: 卓越した思考整理力。複雑な概念が「問題提起→コンテキスト→探求の旅→発見と学び→次のステップ」という明確な流れで整理されている。著者の試行錯誤のプロセスが透明に示されている。

**3.0**: 標準的な思考整理力。基本的な論理構造はあるが、思考プロセスの一部が省略されている。

**1.0**: 非常に弱い思考整理力。断片的な考えが並べられているだけで、論理的なつながりがほとんどない。

#### 実践応用性 (0.0-5.0)
記事の情報が読者の実際の行動や実践にどれだけ役立つかを評価します。

**5.0**: 非常に高い実践応用性。具体的な手順、失敗例とその解決策、つまずきやすいポイントへの対応策を提供している。読者はこの記事だけで実際に行動を起こせる十分な情報を得られる。

**3.0**: 標準的な実践応用性。基本的な実践情報は提供されているが、具体例やステップバイステップの指示が限定的。

**1.0**: 非常に低い実践応用性。情報が断片的で実践に結びつけるのが困難。

#### 構成と読みやすさ (0.0-5.0)
記事の構造、文体、視覚的要素が読者の理解と共感をどれだけ促進するかを評価します。

**5.0**: 卓越した構成と読みやすさ。冒頭の3行で読者の興味を引き、適切な見出し階層で内容が整理されている。短い段落、箇条書き、強調表示が効果的に使われている。

**3.0**: 標準的な構成と読みやすさ。基本的な構造はあるが、一部に長い段落や複雑な説明がある。

**1.0**: 非常に弱い構成と読みやすさ。一貫した構造がなく、文章が冗長で複雑。

#### コミュニケーション力 (0.0-5.0)
記事が読者と共感的につながり、技術情報を人間味を持って伝えているかを評価します。

**5.0**: 優れたコミュニケーション力。読み手の感情を大切にし、個人の経験として共有し、主観的な表現を心がけている。読者の立場に立った情報提供と、共感を呼ぶ語り口で、技術情報に人間味を加えている。

**3.0**: 標準的なコミュニケーション力。基本的な情報は伝わるが、読者との共感的なつながりが限定的。

**1.0**: 非常に弱いコミュニケーション力。読者の存在をほとんど意識していない書き方で、共感や対話の要素がない。

### 評価レポート形式

```markdown
# [記事タイトル] 評価レポート

## 総合評価: [平均点]/5.0

### 防御力: [点数]/5.0
[具体的な所見と例]

### 思考整理力: [点数]/5.0
[具体的な所見と例]

### 実践応用性: [点数]/5.0
[具体的な所見と例]

### 構成と読みやすさ: [点数]/5.0
[具体的な所見と例]

### コミュニケーション力: [点数]/5.0
[具体的な所見と例]

## 総評
[全体的な感想と主な強み]

## 改善提案
[優先的に改善すべき点とその具体的な方法]
```

### 評価の姿勢

- 批判ではなく建設的なフィードバックを心がける
- 著者の経験レベルや記事の目的を考慮して評価する
- 良い点を明確に指摘し、改善点は具体的な提案と共に述べる
- 記事の「学び続ける姿勢」や「思考の過程」としての価値も評価する
- 80%の完成度でも価値があることを認識する

この品質保証ルールは、ブログの成長と共に継続的に改善していくものとします。
